# <img width="27" alt="boolock_logo" src="https://github.com/user-attachments/assets/469d65f4-d33d-46d9-a91c-29443d9080fe"> 프로젝트 소개 <a id="introduce"></a>

### 웹개발 입문자를 위한 HTML과 CSS 블록코딩 플랫폼

```html
<h1>hello world!</h1>
```

웹개발 입문자들에게는 이 한 줄의 코드도 딱딱하고 재미없게 느껴질 수 았습니다. <br/>
이런 사람들이 블록을 조립하며 쉽고 즐겁게 자신만의 웹사이트를 만들 수 있다면 얼마나 좋을까요?

이를 위해 저희는 코딩 입문자를 위한 HTML과 CSS 블록코딩 플랫폼 **BooLock**을 개발하였습니다. <br/>
**BooLock**은 HTML과 CSS를 블록코딩 방식으로 학습할 수 있는 웹 기반 플랫폼으로, 웹 개발을 처음 접하는 입문자들이 쉽게 학습하며 즐길 수 있도록 설계되었습니다. <br/>

특히 저희 팀은 프론트엔드 개발자 5명으로 구성되어 있는 만큼, 완성도 있는 서비스를 제공하기 위해 사용자 경험(UX)을 개선하는데 주력하였습니다.

## FE

### 블록리 라이브러리 커스텀

저희는 블록 코딩 사이트 **BooLock**을 만들기 위해 Google의 **Blockly 라이브러리**를 사용했습니다. Blockly는 코드를 직접 작성하지 않고도 블록을 조립하는 방식으로 프로그래밍할 수 있게 해주는 라이브러리입니다.

저희 기획에 맞게 구현하기 위해서는 라이브러리의 내부 로직을 이해하고 핵심 코드를 수정해야 했습니다. 오른쪽에서 원하는 블록을 선택할 수 있는 메뉴(Toolbox), 실제 블록들, 그리고 이 블록들을 자유롭게 조립할 수 있는 작업 공간(Canvas) 같은 요소들의 동작 방식을 파악한 후, Blockly의 주요 클래스들을 상속받아 저희만의 커스텀 UI를 만든 것이 의미 있는 기술적 경험이었습니다.

   Blockly에서는 이런 작은 블록 하나를 생성하고 동작시키기 위해 다양한 클래스들이 사용됩니다. 블록 객체 인스턴스를 생성하는 **BlockSvg**, 블록 내부 구성 요소(Field) 인스턴스를 생성하는 **Field 관련 클래스들**, 블록 정보를 렌더링하는 **Renderer**, 블록 구성 요소를 정의하고 사이즈 및 좌표를 계산하는 **RenderInfo**, 그리고 블록의 배경을 그려주는 **Drawer** 등 여러 핵심 클래스들이 있습니다.

   블록은 **SVGElement**로 구성되어 있기 때문에, 원하는 모양으로 블록을 커스텀하려면 아래와 같이, 블록 모양 구성을 구성하는 주요 클래스들을 **상속받아 오버라이딩**하는 것이 핵심이었습니다.

   - 사이즈 및 좌표계산을 해주는 RenderInfo 클래스
   - 블록 모양을 그려주는 Drawer 클래스의 메소드를 실행하기 전 SVG의 path를 그려주는 ConstantProvider 클래스
   - 사이즈와 디자인을 동적으로 업데이트 해주는 각 Field클래스들의 update, view 관련 메소드들

   이 클래스들을 오버라이딩 하여 기획한 디자인대로 블록을 커스텀하며 겪었던 트러블 슈팅과 개발과정은 아래 문서들에서 더 자세하게 확인 가능합니다.

   사용자들이 HTML 블록을 쉽게 찾아 선택하고, 필요한 CSS 클래스를 자유롭게 추가할 수 있는 기능을 만들고 싶었습니다. 이를 위해 HTML 블록을 범주별로 조회할 수 있는 탭과 스타일 클래스를 추가하는 탭을 만들고, 각 탭마다 다른 화면을 제공하는 UI를 구성했습니다.

   하지만 Blockly 라이브러리를 사용하면서 두 가지 중요한 제약사항에 있었습니다.

   1. **탭과 입력 기능의 부재:** Blockly는 기본적으로 탭 인터페이스를 지원하지 않았습니다. 또한, CSS 클래스 탭에서는 사용자가 직접 입력한 CSS 클래스명 블록을 생성해야 하지만, Blockly 구조상 flyout에 블록과 특정 태그 외 요소는 추가할 수 없어 `input` 요소를 직접 삽입할 수 없었습니다.
   2. **고정된 Flyout 구조:** Blockly의 블록 목록(Flyout)은 도구상자(Toolbox) 옆에 고정된 형태로만 제공되어, 기획에서 요구했대로 탭별로 다른 flyout을 보여주는 것이 불가능했습니다.

   먼저 탭 기능을 구현하고 flyout의 위치를 수정하기 위해 Blockly가 생성해주는 작업 공간 UI의 DOM 구조를 분석했습니다. Blockly Canvas 내에서 HTML 요소를 사용할 수 있는 유일한 공간이 Toolbox를 감싸는 div였기 때문에, Toolbox 클래스를 상속받아 커스텀 구현을 시작했습니다. DOM 요소 생성 함수를 재정의(override)하여 탭 버튼을 추가하고, 기획에 맞는 디자인을 구현하기 위해 Flyout의 생성 위치를 Toolbox 내부로 이동시켰습니다. 또한, 탭마다 다른 Flyout을 표시할 수 있도록, Blockly 내부에 탭별 Flyout 정보를 등록하고, Flyout 구성을 위한 추가 설정 값을 Toolbox 생성 시 전달 받도록 수정했습니다.

   CSS 클래스 탭 같은 경우에는 사용자 입력을 받아야 했지만, Blockly의 UI가 전부 SVG로 구현되어 있어 input 요소를 추가하기가 까다로웠습니다. SVG 내에 입력 필드를 구현하려면 라이브러리의 핵심 기능을 수정해야 했기 때문에, 대신 Flyout을 div로 감싸고 그 안에 input 요소를 추가하는 방식을 선택했습니다. 이렇게 함으로써 라이브러리와의 호환성을 유지하면서도 필요한 기능을 구현할 수 있었습니다.

3. **스타일 동적 추가하기**
   <div align="center">
      <img src="https://github.com/user-attachments/assets/0a337295-256e-4a59-92a7-9c83ce8a1618" alt="storybook description" width="600"/>

   Blockly의 Flyout 내부에 DOM 요소(input, button)를 동적으로 생성하여 사용자 인터페이스를 구성하고, 이를 통해 CSS 클래스 블록을 입력값에 따라 동적으로 생성하거나 삭제할 수 있도록 설계되었습니다. 사용자는 Flyout에서 클래스명을 입력한 후 버튼 클릭이나 Enter 키를 눌러 블록을 생성하며, 입력값은 유효성 검사를 거쳐 중복 여부를 확인한 뒤, 유효한 경우 접두사(`CSS_`)를 추가한 블록으로 등록됩니다. 생성된 블록은 Flyout에 즉시 반영되어 바로 사용할 수 있습니다. 또한, 블록을 우클릭하면 "블록 삭제" 옵션이 나타나며, 해당 블록과 관련된 모든 인스턴스를 삭제하도록 컨텍스트 메뉴가 구현되었습니다. Reset CSS 체크박스와 툴팁 기능도 추가되어 브라우저 기본 스타일 초기화 여부를 선택하고, 사용자 경험을 개선했습니다.

### 코드 하이라이팅

처음에는 블록으로 생성된 HTML과 CSS 코드를 라이브러리를 활용해 예쁘게 표시했지만, 시안과 약간의 차이가 있었고, 블록과 연동한 하이라이팅 기능을 추가하고자 직접 구현하기로 결정했습니다. 기존 하이라이팅 기능은 라이브러리처럼 다른 프로젝트에서도 쉽게 사용할 수 있도록 별도로 작성해 shared 폴더에 구성했으며, 블록 연동 하이라이팅 기능은 별도로 통합하여 사용했습니다. 각 태그, 속성, 값 등은 정규 표현식 파싱을 통해 span 태그로 각각 감싼 후 클래스를 부여하여 스타일을 제공하였습니다. 또한 새롭게 생성되거나 수정되는 코드에는 동적인 애니메이션을 추가하여 변경된 부분을 사용자가 쉽게 확인할 수 있게 했습니다. Blockly 블록과의 연동을 위해 data-block-id 속성을 추가하여 특정 블록과 관련된 코드를 추적할 수 있게 했습니다. 선택한 블록의 시작점은 blockId를 통해 라인을 식별하고, 블록의 길이는 자식 블록을 포함한 재귀 함수를 이용해 계산했습니다. 최종적으로 사용자가 보는 화면에는 데이터 속성과 하이라이트용 span 태그를 제거한 결과물을 제공하며, UI 애니메이션과 hover 효과도 지원해 직관적인 코드 하이라이팅과 사용자 경험을 구현했습니다.

### FSD 아키텍처

프로젝트를 시작하기 전, 팀 내에서는 기능 단위로 구조화하여 코드의 가독성과 재사용성을 높이는 FSD 구조를 도입하자는 의견이 있었습니다. FSD 방식은 폴더를 세분화하는 만큼 대규모 프로젝트에 적합하다는 생각이 있었지만, 프로젝트를 분할하여 관리할 수 있는 장점이 매력적으로 다가왔습니다. 또한, 이번 프로젝트는 학습의 목적이 크기 때문에 팀원들 모두 새로운 폴더 구조를 적용해보고자 했습니다.

그러나 FSD를 처음 접하다 보니 각 레이어를 어떻게 나눌지에 대한 기준이 명확하지 않아 혼란스러운 부분이 많았습니다. 특히, 팀원마다 기준이 달랐던 **feature 레이어**는 정의가 애매하다는 결론을 내리고, 이를 제외한 구조를 가져가기로 결정했습니다

### 성능 개선 (SEO)

팀원분의 동생이 웹사이트를 써보고 싶어서 인터넷에 검색했는데 사이트가 나오지 않는다고 해서, 사용자 유입을 위해 SEO를 도입해야겠다고 생각했습니다.

React는 SPA 특성상 하나의 `index.html` 파일로 모든 페이지를 렌더링합니다. 이는 각 페이지별로 고유한 메타 정보를 제공하지 못해 검색 엔진 크롤러가 콘텐츠를 제대로 인식하지 못하는 문제를 야기합니다. 이러한 문제를 해결하기 위해 **react-helmet-async**를 도입했습니다. 최상위 파일을 `HelmetProvider`로 감싸고, 각 페이지에 `<Helmet>` 태그를 추가하여 제목과 설명 같은 메타 정보를 설정함으로써 검색 엔진 크롤러가 올바른 정보를 인식하도록 했습니다.
또한, 소셜 미디어에서 공유되는 콘텐츠를 매력적으로 표시하기 위해 **Open Graph** 메타 태그를 추가했습니다. 이를 통해 공유 링크에 제목, 설명, 이미지를 지정해 사용자 경험과 전환율을 향상시켰습니다.

이외에도 이미지 alt 속성, robots.txt, 다양한 메타 태그를 관리하며 SEO 최적화를 진행했고, 그 결과 Lighthouse SEO 점수를 75점에서 100점으로 향상시켰습니다.
